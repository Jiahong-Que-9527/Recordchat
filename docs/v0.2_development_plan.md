# RecordChat v0.2 开发计划

v0.1 已完成端到端 RAG MVP。v0.2 目标：**语义检索增强 + 第一个生态集成钩子 + 前端体验升级**。

详细路线图见 [roadmap.md](roadmap.md)。本文是**可执行的开发步骤**，与 GitHub Milestones / Issues 一一对应。

---

## 推荐开发顺序

```
v0.2.1 Ontology-aware retrieval   ← 先做，提升检索质量，为后续功能打底
    ↓
v0.2.3 AviationLakehouse narrative  ← 可与 2.2 并行；以内容 + domain 层为主
    ↓
v0.2.2 RecordForge integration    ← 需要 connector 模式，依赖 pipeline 稳定
    ↓
v0.2.4 Frontend upgrade           ← 最后做 UI 大改；后端需先有 /chat/stream
```

> **说明**：v0.2.4 的 streaming 后端（Issue #17）可在 v0.2.2 之前独立完成，但完整前端迁移建议放在语义层和生态 narrative 就绪之后，避免 UI 改两次。

---

## v0.2.1 — Ontology-aware retrieval

**目标**：从 `data/raw/ontology/one_record_core.ttl` 解析出类、属性、关系，实现 entity-first 检索，逐步替代纯手工 `one_record_schema.py` 关系表。

### 步骤

| 步骤 | 做什么 | 主要文件 | 完成标志 |
|------|--------|----------|----------|
| 1.1 | 写 ADR，定义 OntologyGraph 数据结构与检索策略 | `docs/adr/0002-ontology-aware-retrieval.md` | ADR merged |
| 1.2 | 实现 ontology parser（rdflib） | `backend/app/domain/ontology_parser.py` | 单元测试：能解析出 ≥N 个 Class/Property |
| 1.3 | 构建 OntologyGraph（类层次、object property 关系） | `backend/app/domain/ontology_graph.py` | `get_related(class)` 返回解析结果 |
| 1.4 | ingest 时将 ontology 关系写入 chunk metadata | `rag/chunker.py`, `rag/ingest.py` | chunk 带 `entity` / `related_entities` |
| 1.5 | 实现 entity-first retriever 策略 | `rag/retriever.py` 或新 `rag/entity_retriever.py` | 问 "Piece" 时优先返回 Piece 相关 chunk |
| 1.6 | pipeline 接入：detect entities → boost/filter chunks | `rag/pipeline.py` | 关系类问题 related_concepts 更准确 |
| 1.7 | 用 ontology 图 augment `one_record_schema`（保留 glossary 降级） | `domain/one_record_schema.py` | 手工表为 fallback，ontology 优先 |
| 1.8 | 扩展 eval + pytest | `data/eval/questions.yaml`, `tests/` | eval 通过；新增 ontology 专项测试 |
| 1.9 | 更新 architecture.md + demo_script | `docs/` | 文档反映 v0.2.1 行为 |

### 验收标准

- [ ] `POST /ingest` 后，ontology 类/属性被结构化索引
- [ ] `"Explain Shipment and Piece"` 检索结果明显优于 v0.1
- [ ] `related_concepts` 部分来自 ontology 解析，不仅手工表
- [ ] 无 API key 时仍完全 offline 可跑
- [ ] 现有 12 条 eval 不退化

## v0.2.2 — RecordForge integration

**目标**：支持「生成 N 条合成 shipment / piece」类请求，通过 RecordForge connector 返回 JSON-LD（初期可用 stub）。

### 步骤

| 步骤 | 做什么 | 主要文件 | 完成标志 |
|------|--------|----------|----------|
| 2.1 | 定义 connector 抽象 + 配置项 | `backend/app/connectors/base.py`, `core/config.py` | `RECORDFORGE_URL` 等 env |
| 2.2 | 新增 query type：`synthetic_data_generation` | `models/chat.py`, `rag/pipeline.py` | classifier 识别 generate N shipments |
| 2.3 | 实现 RecordForge HTTP client（或 stub） | `connectors/recordforge.py` | 返回合法 JSON-LD 列表 |
| 2.4 | pipeline 分支：调 connector → `structured_output` | `rag/pipeline.py` | `/chat` 响应含多对象 JSON-LD |
| 2.5 | 前端展示多 JSON-LD 块（v0.1 UI 最小支持） | `frontend/components/JsonLdViewer.tsx` | 能渲染数组 |
| 2.6 | pytest + eval 新增 2–3 条 | `data/eval/questions.yaml` | synthetic 问题有 expected 结构 |
| 2.7 | 写 ADR + demo 话术 | `docs/adr/`, `docs/demo_script.md` | demo 可演示 RecordForge 流程 |

### 验收标准

- [ ] `"Generate 5 synthetic shipments with pieces"` 返回结构化 JSON-LD
- [ ] RecordForge 不可用时优雅降级（说明 + 单条模板 fallback）
- [ ] connector 与 RAG 解耦，符合 SPEC 分层

## v0.2.3 — AviationLakehouse narrative

**目标**：能清晰回答 ONE Record 数据如何进入 ALH 的 Bronze / Silver / Gold 层（知识 + domain 映射，非 live connector）。

### 步骤

| 步骤 | 做什么 | 主要文件 | 完成标志 |
|------|--------|----------|----------|
| 3.1 | 新增 ALH 知识文档 | `data/raw/one_record_docs/aviation_lakehouse.md` | ingest 可索引 |
| 3.2 | 实现 ALH 映射模块 | `backend/app/domain/alh_mapping.py` | Piece/Shipment → 各层说明 |
| 3.3 | 扩展 classifier / prompt 处理架构类问题 | `rag/pipeline.py`, `rag/prompt.py` | ALH 问题引用映射 + sources |
| 3.4 | glossary 补充 ALH 术语 | `domain/glossary.py` | Bronze/Silver/Gold 有词条 |
| 3.5 | eval + demo 更新 | `data/eval/questions.yaml`, `docs/demo_script.md` | q013+ ALH 问题通过 |
| 3.6 | roadmap/architecture 交叉引用 | `docs/architecture.md` | ALH 叙事可追踪 |

### 验收标准

- [ ] `"How could ONE Record data connect to AviationLakehouse?"` 回答含 Bronze/Silver/Gold
- [ ] sources 指向新增 ALH 文档
- [ ] 仍为 grounded RAG，非纯 LLM 编造

## v0.2.4 — Frontend upgrade (Vercel AI Chatbot)

**目标**：用 [Vercel AI Chatbot](https://github.com/vercel/chatbot) 模板替换 v0.1 手写 UI，支持 streaming，保留 RecordChat 领域组件。

### 步骤

| 步骤 | 做什么 | 主要文件 | 完成标志 |
|------|--------|----------|----------|
| 4.1 | LLM provider 增加 `complete_stream()` | `core/llm.py` | local + OpenAI-compat 可流式 |
| 4.2 | 新增 `POST /chat/stream`（SSE） | `api/chat.py`, `rag/pipeline.py` | token 流 + 末尾 metadata 事件 |
| 4.3 | 保持 `POST /chat` 同步接口 | `api/chat.py` | eval 脚本不受影响 |
| 4.4 | Fork/scaffold Vercel chatbot 模板 | `frontend/` 重构 | `npm run build` 通过 |
| 4.5 | 接入 shadcn/ui + AI Elements | `frontend/components/` | 对话区用 AI SDK 组件 |
| 4.6 | 实现 FastAPI adapter route（AI SDK 协议） | `frontend/app/api/chat/route.ts` | 代理到 backend `/chat/stream` |
| 4.7 | 移植 Sources / Related concepts / JSON-LD viewer | `frontend/components/` | 领域 UI 保留 |
| 4.8 | Markdown + 代码高亮 | 新依赖 `react-markdown` 等 | 回答可读性提升 |
| 4.9 | Docker / env / README 更新 | `docker-compose.yml`, `README.md` | VPS demo 仍可用 |
| 4.10 | 端到端 smoke：5 条 demo 问题 | 手动 + 可选 Playwright | demo_script 全流程 OK |

### 验收标准

- [ ] 回答 token-by-token 流出
- [ ] sources / JSON-LD / related concepts 在流结束后展示
- [ ] `evaluate_rag.py` 仍用同步 `/chat`
- [ ] `docker compose up --build` 一键启动

3. 实现 + 测试
4. PR 关联 issue：`Closes #<n>`
5. merge 后关闭 issue，进入下一个
